## 2. 4-class doc taxonomy

Used as a lightweight decision model when classifying files; not a per-file
label. The four classes are stated once in `docs/_archive/README.md` so
external readers understand archived material's status.

| Class | Maintenance | Examples |
|---|---|---|
| Normative spec | Tracks design decisions; changes require sign-off | `spec/ni/doc/*.md`, `spec/ni/dv/*.md` |
| Maintained guide | Updated whenever underlying code drifts | `README.md`, `docs/architecture.md`, `docs/development.md`, `tests/scenarios/README.md`, `spec/ni/README.md` |
| Generated reference | Tool emits; humans do not edit | `c_model/FEATURE_INVENTORY.md` |
| Historical archive | Frozen; may conflict with current implementation | `docs/_archive/**` |

Normative / generated status is established by directory-level README context
and per-file banners (e.g. `FEATURE_INVENTORY.md` already opens with an
auto-generated banner). The taxonomy is a decision aid, not a tagging system.

## 3. Doc style guide

### Filenames

- New documents: `lowercase-kebab-case.md` (e.g. `cosim-architecture.md`,
  `coding-conventions.md`)
- Module entry points keep `README.md` (project root convention)
- Rationale: matches OpenTitan / FlooNoC; URL-friendly; visually
  distinguishes documentation from C++ `snake_case` source

### Per-class structure templates

| Class | Required sections |
|---|---|
| `docs/architecture.md` (single file) | `# Title` → `## Purpose` → component sections → `## References` |
| `docs/development.md` (single file) | `# Title` → workflow / conventions / generated files / how-to sections → pre-submit checklist → `## References` |
| Module `README.md` | `# Module` → one-sentence summary → `## Build` → `## Test` → `## Layout` → `## See also` |

Section headings inside each doc are chosen pragmatically; the spec does not
mandate identical subsection lists across guides.

### Writing style

- Tone: terse, declarative. Bullets for parallel items, prose for narrative,
  tables for ranges / options
- Voice: imperative for how-tos (`Build the c_model`); descriptive for
  architecture (`The c_model loads scenarios via ...`)
- Headings: sentence case, not Title Case
- One `# Title` per document; all sub-headings `##` or deeper
- Code blocks must carry a language tag (` ```bash `, ` ```cpp `,
  ` ```yaml `)
- No emoji (per project memory: only use emoji when explicitly requested)
- Cross-links use relative paths with `.md` suffix: `[label](../arch/cosim.md)`
- Per-document length budget: 200-400 lines; split if exceeded
- Character encoding: UTF-8 without BOM
- ASCII only — no `→`, `←`, `⇒`, `⇐` (Windows GBK locale produces mojibake);
  use `->`, `<-`, `=>`, `<=`

### Lint

- `markdownlint-cli2` for general Markdown hygiene
- The ASCII-only rule is enforced via a custom check in
  `tools/lint_scenarios.py` (or a sibling script) — to be added during plan
  execution if cheap

### Out of scope

- `spec/ni/doc/*.md` and `spec/ni/dv/*.md` retain their existing conventions
  (mixed casing, varying structure). They are normative spec; cosmetic
  restyling is not worth the invariant risk.

## 1. System context (~30 lines)
- Project purpose: AXI4 NoC NI behavioural model + Verilator cosim
- ASCII overview diagram (AXI Master ─► NMU ─► NoC ─► NSU ─► AXI Slave)
- Boundary definitions (AXI pin, NoC flit)
- Two NI implementations (behavioural c_model + future RTL), behaviorally
  equivalent at AXI and NoC pin boundaries
- Pointer to spec/ni/README.md as the normative spec

## 2. NI components — NMU, NSU, router (~60 lines)
- NMU role, RoB, QoS, address remapping; refs to AMD PG313 / PG406
- NSU role and asymmetry (no RoB)
- Uniform router; XY routing
- AXI endpoints (AxiMaster, AxiSlave, Memory) cited as cocotbext-axi
  derivatives (link to ATTRIBUTION.md)
- Source path links per component

## 3. c_model component flow and tick discipline (~90 lines)
[Merged from previous outline §3 and §4]
- Component map: AxiMaster -> AxiSlavePort -> Nmu -> LoopbackNoc -> Nsu ->
  AxiMasterPort -> AxiSlave + Memory
- β tick discipline (1 tick per cycle; inputs first, then outputs)
- Hermetic singleton; data flow only via IO structs
- Dual flow control (Valid/Ready and Credit-based; compile-time selected)
- Factory pattern; RTL parameter analogy
- Hot-swap Router_Interface / NI_Interface abstract base classes
- Pipeline boundaries explicit; no combinational delays across components

## 4. Cosim and Verilator boundary (~80 lines)
- Stage 5b wire-wrap: each c_model component lives in an SV shell module;
  cross-component data flow only via registered SV wires
- DPI 3-step pattern per shell (set_inputs → tick → get_outputs)
- 5 shells: AxiMaster, Nmu, LoopbackNoc, Nsu, AxiSlave
- wb2axip protocol checkers bind on two AXI bundles
- Vtb_top binary; runtime plusarg scenario load
- wb2axip slave structural limits (absorbed from cosim/KNOWN_LIMITATIONS.md
  active content):
  - Single-beat write (AWLEN = 0)
  - Single outstanding write (wr_pending ≤ 1)
  - AW/W strict in-order
  - No exclusive monitor; no DECERR address-map check
- wb2axip is a formal-verification slave; structural limits are temporary
- `cosim/tests/wb2axip_block.hpp` — runtime predicate that maps scenario
  content to a SKIP reason

## 5. Verification layers (~60 lines)
- Run-all tests: c_model/tests/axi/test_integration.cpp and
  cosim/tests/test_cosim_integration.cpp; both consume
  kAllAxi4Scenarios (generated by CMake glob over
  tests/scenarios/AX4-*/scenario.yaml)
- Cosim runtime skip path
- Scoped tests: port_pair_loopback, request_response_loopback,
  checker_fires_on_violation; use RequireKnownScenario to abort on stale
  IDs
- Scenario tree convention AX4-CAT-NNN_slug; full taxonomy in
  tests/scenarios/README.md

## 6. AXI4 conformity scope (~30 lines)
- Covered IHI 0022H sections (§A3 handshake/burst/response, §A5/§A6
  ordering, §A7.2.4 exclusive)
- Deliberately excluded (AXI3 features, AXI4 atomics, ACE, AXI5, REGION,
  multi-master / bus matrix, CDC)
- Future phases: full sideband, multi-master, BFM replacement of wb2axip

## References (~10 lines)
- spec/ni/README.md
- spec/ni/doc/protocol_rules.md
- spec/ni/doc/packet_format.md
- tests/scenarios/README.md
- c_model/FEATURE_INVENTORY.md
- c_model/include/axi/ATTRIBUTION.md
```
